# جاهزية Roadly للإنتاج (Production Readiness)

## الخلاصة: **ليس جاهزاً بعد للإنتاج**

المشروع جاهز من ناحية **الهيكلة والتصميم والتجربة**، لكن ينقصه ربط بالباكند، بناء للإنتاج، وأمور أمنية/تشغيلية قبل النشر الفعلي.

---

## ✅ ما هو جاهز (Ready)

| البند | الحالة |
|-------|--------|
| هيكل المشروع (Clean Architecture) | ✅ |
| نظام التصميم (ألوان، خطوط، مكونات) | ✅ |
| شاشات Auth (ترحيب، تسجيل دخول، إنشاء حساب) | ✅ واجهات فقط |
| حفظ وعرض المعلومات المحفوظة | ✅ (ويب: localStorage؛ موبايل: خلال الجلسة) |

| TanStack Query + Zustand | ✅ |
| Location permission + خدمة الموقع | ✅ |

| Error Boundary + Toast + Loader عام | ✅ |

## ❌ ما ينقص قبل البرودكشن

### 1) الباكند والـ Auth الحقيقي
- **تسجيل الدخول والتسجيل** حالياً وهميان (toast "غير متوفر" ثم الانتقال للرئيسية).
- مطلوب: ربط مع API حقيقي (تسجيل، دخول، refresh token)، وحفظ الجلسة (مثلاً AsyncStorage على الموبايل).

### 2) بناء وتوزيع التطبيق (Build & Deploy)
- لا يوجد **EAS Build** أو إعداد لبناء أندرويد/آيفون للإنتاج.
- مطلوب: إعداد `eas.json` وبناء production (واختيارياً التوزيع عبر EAS Submit أو المتاجر).

### 3) متغيرات البيئة (Environment)
- لا يوجد ملف `.env` أو `.env.production`؛ الـ API يعتمد على `EXPO_PUBLIC_API_URL` أو localhost.
- مطلوب: تعريف `EXPO_PUBLIC_API_URL` للإنتاج وعدم وضع أسرار داخل الكود.

### 4) التخزين المحلي على الموبايل
- **المعلومات المحفوظة** (آخر بريد دخول/تسجيل) تعمل على الويب عبر `localStorage`؛ على الموبايل تُحفظ في الذاكرة فقط (تضيع عند إغلاق التطبيق).
- مطلوب: استخدام **AsyncStorage** (أو بديل آمن) على الموبايل لحفظ هذه البيانات.

### 5) الخريطة والمزودون
- شاشة الخريطة placeholder؛ لا توجد واجهة كاملة (علامات، قائمة أقرب مزودين، تفاصيل مزود).
- لا يوجد ربط فعلي مع API `/providers/nearby` (الـ backend غير متوفر أو غير مكتمل).

### 6) الأدوار (Roles) والصلاحيات
- الأنواع والثوابت معرّفة (User, Mechanic, Tow, Rental, Admin) لكن لا يوجد توجيه فعلي للشاشات حسب الدور ولا حماية للمسارات.

### 7) الجودة والمراقبة
- لا توجد **اختبارات** (Unit / Integration).
- لا يوجد **تسجيل أخطاء** (مثل Sentry) ولا تحليلات استخدام.

### 8) متطلبات المتاجر
- **Google Play**: يحتاج سياسة خصوصية، حساب مطور، وإعدادات التوقيع.
- **App Store**: يحتاج حساب Apple Developer، سياسة خصوصية، وإعدادات التوقيع والشهادات.

## خطوات مقترحة للوصول للبرودكشن

1. **ربط Auth بالباكند**: استدعاء APIs تسجيل/دخول حقيقية، حفظ token وجلسة، وربط الـ HTTP client بالـ token.
2. **إعداد EAS Build**: إضافة `eas.json` وبناء production لـ Android و iOS.
3. **إعداد البيئة**: إنشاء `.env.production` و`EXPO_PUBLIC_API_URL` لسيرفر الإنتاج.
4. **استخدام AsyncStorage** على الموبايل لحفظ آخر بريد دخول/تسجيل (واختيارياً الجلسة).
5. **إكمال شاشة الخريطة وقائمة المزودين** وربطها بـ `/providers/nearby` عند جاهزية الباكند.
6. **توجيه حسب الدور**: عرض شاشات مختلفة حسب (User / Mechanic / Admin...) وحماية المسارات.
7. **(اختياري)** إضافة اختبارات أساسية وتكامل مع Sentry (أو غيره) لتسجيل الأخطاء.

بعد تنفيذ الخطوات أعلاه ووجود باكند مستقر، يمكن اعتبار التطبيق **جاهزاً للنشر الأولي (برودكشن)**.
