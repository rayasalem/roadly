# MechNow – تقرير الاختبار الوظيفي والخرائط (Web & Mobile)

**التاريخ:** 2026-03  
**النطاق:** ويب + موبايل (تدقيق الكود + تتبع التدفقات)  
**الخلفية:** Node/Express (منفذ 4000)  
**الواجهة:** React Native / Expo (ويب 8082)

---

## 1. الملخص التنفيذي

| المجال | الحالة | ملاحظات |
|--------|--------|---------|
| الأزرار والصفحات (كل الأدوار) | ✅ يعمل | جميع الشاشات والتنقل والـ CTAs مربوطة |
| تدفق طلب الخدمة (من الطلب حتى الإكمال) | ✅ يعمل | أقرب مزود، لوحة المزود، تحديث الحالة، سجل الطلبات |
| الخريطة والمزودون | ✅ يعمل | OSM، اسم/صورة/هاتف/موقع، بحث مدينة/قرية (بيانات اختبار) |

## 2. اختبار الأزرار والصفحات (كل الأدوار)

### 2.1 العميل (User)
- **الخريطة (أول شاشة):** شريط بحث (عنوان + منطقة)، فلاتر (الكل / ميكانيكي / ونش / تأجير)، "موقعي"، ماركرات المزودين، Bottom Sheet (طلب خدمة / فتح الخريطة)، BottomNavBar. **كلها نشطة.**
- **البحث بالمدينة/القرية:** عند عدم وجود مفتاح Google Places تُعرض **بيانات اختبار** (بزارية، نابلس، رام الله، جنين، طولكرم، الخليل، دبي) مع إحداثيات حقيقية؛ اختيار مكان يحرّك مركز الخريطة.
- **الرئيسية:** بدء الرحلة، أقرب المواقع، الأماكن المحفوظة، التنقل إلى الخريطة/الملف/الدردشة/الإشعارات/الإعدادات. **نشط.**
- **طلب الخدمة:** إنشاء طلب (ميكانيكي/ونش/تأجير)، تتبع الحالة، تقييم عند الإكمال. **مربوط بالـ API.**
- **طلباتي (سجل الطلبات):** من الملف الشخصي → "طلباتي" → قائمة الطلبات مع الحالة والتاريخ؛ النقر يفتح تفاصيل الطلب. **مربوط بـ GET /requests/customer.**
- **الملف الشخصي / الإعدادات / الدردشة / الإشعارات:** التنقل والإجراءات (مثل تعليم كمقروء) **تعمل.**

### 2.2 الميكانيكي (Mechanic)
- **لوحة التحكم:** إحصائيات (مهام اليوم، في الطريق، التقييم)، فلتر (الكل / جديد / في الطريق / في الكراج)، بطاقات الطلبات مع **قبول / رفض** (PATCH /requests/:id/status)، زر الخريطة، الملف. **نشط.** الطلبات المعينة له تظهر بعد أن يختار المستخدم هذا المزود من الخريطة.

### 2.3 الونش (Tow)
- **لوحة التحكم:** نفس نمط الميكانيكي؛ قبول/رفض، الطلبات المعينة تظهر. **يعمل.**

### 2.4 تأجير السيارات (Car Rental)
- **لوحة التحكم:** إحصائيات، مركبات، حجوزات. الـ API قد يعيد بيانات فارغة؛ الواجهة تعرض بدون خطأ.

### 2.5 الأدمن (Admin)
- **لوحة التحكم:** تبويبات (ميكانيكي / ونش / تأجير)، إحصائيات، لوحات الطلبات، "عرض القائمة" → AdminProviderList. **الأزرار والتنقل تعمل.**

## 3. تدفق طلب الخدمة (End-to-End)

### 3.1 إنشاء الطلب من قبل المستخدم
1. المستخدم يفتح **الخريطة** → يختار مزودًا → **طلب خدمة** → الانتقال إلى **طلب الخدمة** مع `serviceType` + `providerId`.
2. **RequestScreen** يستخدم `useUserLocation` للموقع، ويستدعي `createMutation.mutate({ serviceType, origin, providerId })` → **POST /requests** (يتطلب تسجيل دخول).
3. عند النجاح يُعيَّن `requestId` ويظهر **الطلب الحالي** (ID، الحالة). عند 401 يُوجَّه المستخدم لتسجيل الدخول.

**الحالة:** ✅ يعمل عند تسجيل الدخول.

### 3.2 تخزين الطلب ورؤيته
- **الخلفية:** الطلب يُحفظ في `requestStore` مع `customerId` واختياريًا `providerId` و`status: 'pending'`.
- **المستخدم:** شاشة **طلباتي** تعرض قائمة الطلبات (GET /requests/customer). النقر على طلب يفتح **RequestScreen** مع `requestId` لعرض التفاصيل والحالة.
- **المزود:** **GET /dashboard/mechanic** و **GET /dashboard/tow** يعيدان الطلبات **المعينة لهذا المزود** + الطلبات المعلقة غير المعينة، لذلك يرى الطلب عند اختيار المستخدم له من الخريطة.

### 3.3 تحديث الحالة (مقبول، قيد التنفيذ، مكتمل)
- **المزود:** قبول → PATCH `status: 'accepted'`؛ في الطريق → `on_the_way`؛ إكمال → `completed`. الخلفية تحدّث المخزن وتنشئ **إشعارات** للعميل.
- **المستخدم:** RequestScreen يستخدم `useRequest(requestId)` (GET /requests/:id). الحالة ووقت الوصول (ETA) تظهر بشكل صحيح؛ إعادة فتح الشاشة أو إبطال الاستعلام يعيد التحديث.

**الخلاصة:** التدفق يعمل من الطلب حتى الإكمال؛ شاشة طلباتي + فتح الطلب بالـ requestId مكتمل.

## 4. الخريطة والمزودون

### 4.1 العرض
- **الويب:** OSM عبر Leaflet؛ طبقات، موقع المستخدم (دائرة)، ماركرات المزودين (لون حسب الدور)، نافذة منبثقة تحتوي **الاسم، الصورة (أو placeholder)، الهاتف، "طلب خدمة / اتصل"**.
- **الموبايل (Native):** react-native-maps مع ستايل Silver، ماركرات، Polyline (المستخدم → الوجهة)، Bottom Sheet بمعلومات المزود.
- **البيانات:** مزودون من **API /providers/nearby** أو **قائمة fallback** (6 مزودين تجريبيين) عند فشل الـ API أو 401؛ الخريطة لا تبقى فارغة.

### 4.2 أقرب مزود
- **useSortedNearbyProviders:** يحسب موقع المستخدم، يجلب المزودين القريبين، يصفّيهم حسب **Haversine** (الأقرب أولاً). الخريطة والبطاقة السفلية تعرضان أقرب مزود عند توفر البيانات.

### 4.3 البحث بالمدينة والقرية
- **بدون مفتاح Google:** `placesApi` يعيد **TEST_PLACES** (بزارية، نابلس، رام الله، جنين، طولكرم، الخليل، دبي) مع إحداثيات. المستخدم يكتب في شريط البحث ويختار مكانًا فيُحدَّث مركز الخريطة.
- **مع مفتاح Google:** Autocomplete + Place Details كما هو معتاد.

### 4.4 النقر على المزود
- عرض التفاصيل في Popup (ويب) أو Bottom Sheet (موبايل)، مع خيار **طلب خدمة** ينتقل إلى RequestScreen مع `serviceType` و`providerId`.

**المشاكل:** لا توجد. الاسم، الصورة، الهاتف، والموقع معروضون؛ الموقع من API/mock (بدون push فوري).

## 5. API والتفويض

| Endpoint | التفويض | السلوك |
|----------|---------|--------|
| POST /requests | مطلوب | 401 عند عدم/فساد التوكن → توجيه لتسجيل الدخول |
| GET /requests/:id, PATCH /requests/:id/status | مطلوب | نفس السلوك |
| GET /requests/customer | مطلوب | قائمة طلبات العميل (سجل الطلبات) |
| GET /requests/provider | مطلوب | طلبات المزود |
| GET /providers/nearby | اختياري | 401 → الواجهة تستخدم **fallback** (قائمة تجريبية) |
| GET /notifications | مطلوب | 401 → توجيه لتسجيل الدخول |
| GET /dashboard/mechanic, /dashboard/tow | مطلوب | Mock مدعوم في التطوير؛ 401 عند عدم التوكن |
| POST /notifications/unregister | اختياري | 204 |

**الخلاصة:** التفويض متسق. في وضع التطوير، التوكنات التجريبية (mock-access-*) مقبولة من الخلفية فلا 401 غير متوقعة للمستخدمين التجريبيين.

## 6. الإشعارات
- **القائمة:** GET /notifications → NotificationsScreen.
- **تعليم كمقروء:** PATCH /notifications/:id/read.
- **الخلفية:** إنشاء إشعار عند تغيير حالة الطلب (accepted, on_the_way, completed)؛ العميل يستلم الإشعار.

**الخلاصة:** تدفق الإشعارات والـ API صحيحان.

## 7. معالجة الأخطاء والواجهة
- **المزودون:** فشل جلب المزودين → عرض **قائمة fallback** + رسالة "عرض بيانات مخزنة" عند الحاجة.
- **إنشاء الطلب:** `createMutation.isError` → عرض الرسالة تحت الزر.
- **الشاشات:** ErrorWithRetry (خريطة، إشعارات، طلباتي). Toasts للنجاح/الخطأ.
- **عام:** UnauthorizedHandler يوجّه لصفحة تسجيل الدخول عند 401 (ما عدا /providers/nearby).
- **التصميم:** SafeArea، AppHeader، BottomNavBar، ثيمات الأدوار، نمط الخريطة Silver؛ تحذير `pointerEvents` من مكتبة @gorhom/bottom-sheet وليس من كود التطبيق.

## 8. ما تم تنفيذه في هذا التدقيق

### 8.1 شاشة طلباتي (Request History)
- **الملفات:** `RequestHistoryScreen.tsx`, `useRequestHistory.ts`, `fetchCustomerRequests()` في requestApi، ENDPOINTS.requestsCustomer.
- **التنقل:** من الملف الشخصي (عميل) → زر "طلباتي" → RequestHistory؛ النقر على طلب → RequestScreen مع `requestId` لعرض التفاصيل والحالة دون نموذج إنشاء جديد.

### 8.2 فتح طلب موجود من الطلبات (requestId في Request)
- **CustomerStackParamList:** Request يقبل `requestId?: string` اختياريًا.
- **RequestScreen:** عند وجود `requestId` في الـ params يُعرض الطلب الحالي فقط (بدون نموذج إنشاء) ويُحمَّل التفاصيل من GET /requests/:id.

### 8.3 بيانات اختبار المدينة/القرية
- **placesApi.ts:** عند عدم وجود `EXPO_PUBLIC_GOOGLE_PLACES_KEY`، `fetchPlaceSuggestions` يعيد **TEST_PLACES** (بزارية، نابلس، رام الله، جنين، طولكرم، الخليل، دبي) مع إحداثيات. `fetchPlaceCoordinates` يحلّ الـ id من TEST_PLACES. البحث والاختيار يعملان بدون Google.

### 8.4 لوحة المزود تعرض الطلبات المعينة (سابقًا)
- **backend/src/routes/dashboard.ts:** GET /dashboard/mechanic و /dashboard/tow يدمجان طلبات هذا المزود (myJobs) + الطلبات المعلقة غير المعينة (pendingJobs).

## 9. توصيات اختيارية
1. **تحديث فوري للحالة:** بعد تحديث المزود للحالة، إبطال استعلام الطلب للعميل (أو polling) حتى يرى RequestScreen الحالة دون إعادة فتح.
2. **pointerEvents:** انتظار تحديث @gorhom/bottom-sheet لاستخدام `style.pointerEvents` أو تعديل node_modules فقط عند الحاجة.
3. **اختبار E2E:** تشغيل Cypress (ويب) أو Detox (موبايل) لتغطية كاملة للمتصفح/الأجهزة.
