# تفاصيل المشروع — ما يعمل وما لا يعمل (Project Status & Features)

آخر تحديث بناءً على الكود الحالي. يوضح كل منطقة: هل مربوطة بالـ API أم mock، وماذا يتوقع أن يعمل أو لا يعمل.

---

## 1. المصادقة (Auth)

| الوظيفة | المصدر | متوقع أن يعمل | متوقع ألا يعمل |
|---------|--------|----------------|-----------------|
| **تسجيل الدخول (Login)** | API حقيقي `POST /auth/login` | عند تشغيل الباكند وإدخال بريد وكلمة مرور صحيحة: يدخل للتطبيق ويحفظ الجلسة (AsyncStorage) | إذا الباكند مغلق أو الـ URL خاطئ: يظهر خطأ شبكة (toast مترجم). إذا بيانات خاطئة: رسالة خطأ من السيرفر |
| **إنشاء حساب (Register)** | API حقيقي `POST /auth/register` | عند تشغيل الباكند: إنشاء حساب ثم دخول تلقائي | نفس سلوك Login عند غياب الباكند أو خطأ في البيانات |
| **الدخول السريع (Mock Login)** | واجهة فقط — لا يمر بالـ API | أزرار (ميكانيكي، ونش، تأجير، أدمن، مستخدم): تسجيل دخول فوري بدون سيرفر وتوجيه للداشبورد المناسب | لا يوجد — يعمل دائماً محلياً |
| **تحديث الجلسة (Refresh Token)** | API + `refreshAccessTokenSafe` | عند انتهاء الـ access token: طلب refresh تلقائي وحفظ الجلسة | إذا الـ refresh فشل أو الباكند لا يدعم الـ endpoint: يتم تسجيل الخروج وتوجيه لشاشة الدخول |
| **تسجيل الخروج (Logout)** | يستدعي `POST /auth/logout` ثم ينظف محلياً | يمسح الجلسة ويوجّه لشاشة الدخول. يعمل حتى لو الباكند رجع خطأ (يُكمل التنظيف المحلي) | لا يوجد سلوك معطّل |
| **استمرار الجلسة (Hydrate)** | AsyncStorage + refresh إن وُجد | عند إعادة فتح التطبيق: استعادة الدخول تلقائياً إذا الجلسة محفوظة وصالحة | إذا الـ refresh انتهى أو فشل: يتم الخروج تلقائياً |

**ملاحظة:** يجب أن يكون `EXPO_PUBLIC_API_URL` في `.env` مضبوطاً (مثلاً `http://localhost:4000`). إذا غير موجود التطبيق يرمي خطأ عند التحميل.

## 2. الخريطة (Map)

| الوظيفة | المصدر | متوقع أن يعمل | متوقع ألا يعمل |
|---------|--------|----------------|-----------------|
| **قائمة المزودين على الخريطة** | **ويب:** `useNearbyProviders` (API) + fallback `MOCK_PROVIDERS` إذا فشل الطلب أو رجع فارغ. **موبايل:** نفس الـ API بدون fallback | **ويب:** دائماً تظهر ماركرات (إما من API أو من 3 مزودين تجريبيين). **موبايل:** تظهر ماركرات فقط إذا الباكند يعمل ويرجع `/providers/nearby` | **موبايل:** إذا الباكند مغلق أو لا يوجد endpoint: الخريطة تظهر بدون أي ماركرات |
| **فلترة (ميكانيكي / ونش / تأجير)** | واجهة + نفس طلب API مع `role` | تغيير الفلتر يحدّث قائمة المزودين إذا السيرفر يدعم `role` في الاستعلام | إذا الـ API لا يفلتر حسب الدور: الفلتر يغيّر الطلب لكن النتيجة تعتمد على الباكند |
| **موقع المستخدم** | `expo-location` (الجهاز) | يظهر موقع المستخدم على الخريطة بعد السماح بالصلاحية | إذا المستخدم منع الموقع أو الجهاز لا يدعم: لا يظهر موقع المستخدم |
| **البحث (Places)** | `usePlacesSearch` + Google Places (مؤسس لكن غير مكتمل) | واجهة البحث موجودة؛ الإكمال يعتمد على ربط Places API ومفتاح Google | البحث عن عنوان لا يحرّك الخريطة فعلياً بدون إكمال الـ integration |
| **نافذة المزود (Bottom Sheet)** | بيانات من الـ `providers` المعروضة | عند الضغط على ماركر: تظهر تفاصيل المزود (اسم، دور، تقييم، حالة) وأزرار "فتح الخريطة" و"طلب خدمة" | الأزرار تعمل (التنقل وطلب الخدمة) لكن "طلب خدمة" يوجّه لشاشة الطلب فقط — إنشاء الطلب الفعلي يعتمد على شاشة الطلب والـ API |
| **خريطة الويب** | Google Maps JS SDK | إذا `EXPO_PUBLIC_GOOGLE_MAPS_KEY` موجود في `.env`: خريطة تفاعلية حقيقية | إذا المفتاح غير موجود: يظهر placeholder نصي (رسالة واضحة بالعربي/الإنجليزي) |
| **خريطة الموبايل** | `react-native-maps` | تعمل بدون مفتاح Google إضافي (خرائط الجهاز) | تحتاج صلاحية موقع وتشغيل الباكند لرؤية المزودين |

## 3. الداشبوردات (Role Dashboards)

كل الداشبوردات (ميكانيكي، ونش، تأجير، أدمن) **تعتمد بالكامل على بيانات تجريبية (mock)** — لا استدعاءات API للقوائم أو الإحصائيات.

| الداشبورد | الوظيفة | متوقع أن يعمل | متوقع ألا يعمل |
|------------|---------|----------------|-----------------|
| **ميكانيكي** | إحصائيات، طلبات، "من طلبني"، فلتر (الكل / جديد / في الطريق / في الكراج)، قبول/رفض، فتح الخريطة، خدماتي/مهاراتي | كل الأزرار والفلترة والتنقل تعمل؛ القبول/الرفض يحدّث الواجهة فقط (mock) | لا يحفظ القبول/الرفض في سيرفر؛ لا بيانات حقيقية |
| **ونش** | نفس الفكرة مع طلبات السحب والمركبات | نفس السلوك | نفس القيود |
| **تأجير** | مركبات، حالة (متاح/مؤجر/صيانة)، حجوزات، إضافة مركبة (mock) | كل التفاعلات محلية | لا مزامنة مع سيرفر |
| **أدمن** | تبويبات (ميكانيكي/ونش/تأجير)، إحصائيات، قوائم، "فتح الخريطة"، "تعديل"، "إدارة المستخدمين" | التنقل والفلترة وفتح الخريطة وتعديل المستخدمين والخدمات في الواجهة (mock) | كل التعديلات محلية؛ لا حفظ في API |

## 4. البروفايل (Profile)

| الوظيفة | المصدر | متوقع أن يعمل | متوقع ألا يعمل |
|---------|--------|----------------|-----------------|
| **بروفايل المزود (ميكانيكي/ونش/تأجير)** | `useProviderProfile` — mock | عرض الاسم، التقييم، الحالة، قائمة الخدمات؛ إضافة/حذف خدمات من قائمة (تحديث واجهة فوري) | التعديلات لا تُرسل لسيرفر؛ تُفقد عند تحديث الصفحة |
| **بروفايل المستخدم/أدمن** | واجهة بسيطة | عرض الاسم والمعلومات الأساسية | لا تحرير معقد أو ربط API |
| **تحرير الخدمات (Bottom Sheet)** | نفس الـ hook (mock) | فتح الشيت، اختيار/إلغاء خدمات، حفظ يحدّث القائمة في الذاكرة | لا حفظ دائم أو API |

## 5. إدارة المستخدمين والخدمات (Admin)

| الوظيفة | المصدر | متوقع أن يعمل | متوقع ألا يعمل |
|---------|--------|----------------|-----------------|
| **قائمة المستخدمين** | `useAdminUsers` — mock | عرض مستخدمين تجريبيين مع دور وحالة وعدد الخدمات المعينة | لا جلب من API؛ القائمة ثابتة من الكود |
| **تعديل مستخدم (اسم، دور، حالة)** | mock في الـ hook | التعديل يظهر فوراً في الواجهة | لا يُحفظ في سيرفر |
| **تعديل خدمات المستخدم** | mock في الـ hook | فتح Bottom Sheet، تفعيل/إلغاء خدمات حسب الفئة، حفظ يحدّث الواجهة | لا إرسال لـ API |

## 6. طلبات الخدمة (Requests)

| الوظيفة | المصدر | متوقع أن يعمل | متوقع ألا يعمل |
|---------|--------|----------------|-----------------|
| **إنشاء طلب (Request)** | API `POST /requests` عبر `requestApi.createRequest` | إذا الباكند يعمل والشاشة مربوطة: إنشاء طلب حقيقي | إذا الباكند مغلق أو الـ endpoint غير مكتمل: خطأ للمستخدم |
| **عرض طلب بالمعرف** | API `GET /requests/:id` | يعمل إذا الباكند يوفّر الـ endpoint | نفس الاعتماد على الباكند |
| **تحديث حالة الطلب** | API `PATCH /requests/:id/status` | قبول/رفض/تحديث حالة من الواجهة إذا الشاشة تستدعي الـ API | يعتمد على ربط الشاشات بـ `requestApi` |

ملاحظة: دواشبورد الميكانيكي/الونش حالياً تستخدم mock للقبول/الرفض ولا تستدعي `updateRequestStatus` من الـ API.

## 7. الإشعارات والـ HTTP

| الوظيفة | المصدر | متوقع أن يعمل | متوقع ألا يعمل |
|---------|--------|----------------|-----------------|
| **إشعارات الدفع (Push)** | تسجيل/إلغاء مع الباكند | إذا الباكند يدعم `/notifications/register` و`/unregister`: تسجيل الجهاز وإلغاؤه عند الخروج | إذا الـ endpoint غير موجود: قد يحدث خطأ صامت أو يُتجاهل |
| **عرض أخطاء الشبكة/API** | `HttpEventsBinder` + i18n | أي فشل طلب (شبكة، وقت، سيرفر): toast برسالة مترجمة (عربي/إنجليزي) | رسائل 401 لا تظهر كـ toast (يتم التعامل معها بتسجيل خروج) |
| **401 وتجديد الجلسة** | `httpClient` interceptors | عند 401: محاولة refresh تلقائية؛ إذا نجحت متابعة الطلب | إذا فشل الـ refresh: مسح الجلسة وتوجيه للدخول (لا toast للـ 401) |

## 8. الواجهة والتصميم

| البند | الحالة |
|-------|--------|
| **i18n (عربي / إنجليزي)** | يعمل — النصوص الرئيسية ورسائل الأخطاء من `strings.ts` |

## 9. ملخص سريع

| المنطقة | يعمل بدون باكند | يعمل مع باكند |
|---------|------------------|----------------|
| **الدخول السريع (Mock)** | ✅ | ✅ |
| **الدخول/التسجيل الحقيقي** | ❌ (خطأ شبكة) | ✅ |
| **الخريطة (ويب)** | ✅ ماركرات تجريبية + placeholder إذا لا مفتاح خرائط | ✅ مزودين من API إن وُجد |
| **الخريطة (موبايل)** | ✅ خريطة بدون ماركرات (ما لم يُضف fallback لاحقاً) | ✅ ماركرات من API |
| **داشبوردات الأدوار** | ✅ كلها mock | نفس الواجهة؛ يمكن لاحقاً ربطها بالـ API |
| **البروفايل وتحرير الخدمات** | ✅ mock | نفس الواجهة؛ يمكن ربط الحفظ بالـ API |
| **أدمن — المستخدمون والخدمات** | ✅ mock | نفس الواجهة؛ يمكن ربطها بالـ API |
| **إنشاء/تحديث طلب خدمة** | ❌ | ✅ إذا الباكند يوفّر الـ endpoints |

## 10. متطلبات التشغيل المحلي

- **Frontend:** وجود `EXPO_PUBLIC_API_URL` في `.env` (مثلاً `http://localhost:4000`). بدونها التطبيق لا يبدأ.
- **خريطة الويب:** إضافة `EXPO_PUBLIC_GOOGLE_MAPS_KEY` في `.env` لخرائط Google تفاعلية؛ بدونها يظهر placeholder.
- **Backend:** تشغيل سيرفر Node.js على نفس الـ URL الموجود في `.env` حتى تعمل تسجيل الدخول والتسجيل والمزودين وطلبات الخدمة.

استخدم هذا الملف كمرجع عند الاختبار أو عند ربط أي جزء بالـ API لاحقاً.
